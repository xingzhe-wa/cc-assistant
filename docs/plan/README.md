## 检查点机制

本项目采用 Harness Engineering 检查点机制，每个检查点代表一个重要的项目里程碑。

### 检查点列表

| 检查点 | 名称 | 状态 | 验收标准 |
|-------|-----|------|---------|
| **CP-0** | 项目初始化 | 🟡 进行中 | 项目骨架完成，依赖配置正确 |
| **CP-1** | 基础框架 | ⏸️ 未开始 | ToolWindow 可显示，Daemon 可启动 |
| **CP-2** | 会话功能 | ⏸️ 未开始 | 完整对话流程，流式输出正常 |
| **CP-3** | 设置功能 | ⏸️ 未开始 | Provider 切换，Agent 管理正常 |
| **CP-4** | 集成功能 | ⏸️ 未开始 | Diff 审查，快捷键正常 |
| **CP-5** | 优化测试 | ⏸️ 未开始 | 性能达标，测试通过 |

### 状态说明

- 🟡 **进行中**: 当前正在执行的检查点
- ⏸️ **未开始**: 等待前置检查点完成
- ✅ **已完成**: 检查点已通过验收
- ❌ **失败**: 检查点验收失败，需要修复

---

## 执行原则

### 1. 顺序执行
检查点必须按顺序执行，后一个检查点依赖前一个检查点的完成。

### 2. 验收驱动
每个检查点都有明确的验收标准，必须全部通过才能进入下一个检查点。

### 3. 文档对照
每个任务都明确引用 `docs/dev/` 下的设计文档，实现必须符合文档规范。

### 4. 进度追踪
项目进度记录在 `00-项目进度.md`，每次对话都会注入上下文。
