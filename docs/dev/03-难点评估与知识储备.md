### 2.2 NDJSON 流式解析

#### 难点描述

```
┌─────────────────────────────────────────────────────────────┐
│  NDJSON 流式解析问题                                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  输入流 (可能跨行)                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ {"type":"chunk","content":"这段代码有问题..."}       │   │
│  │ {"type":"thinking","content":"让我分析一下..."}      │   │
│  │ {"type":"tool_use","toolName":"read_file",          │   │
│  │  "toolInput":{"path":"src/App.kt"}}                 │   │
│  │ {"type":"complete","result":{...}}                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓                                   │
│  解析难点                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 1. 流式读取，可能读取不完整                          │   │
│  │ 2. JSON 跨行，需要缓存和重组                         │   │
│  │ 3. 标签识别，需要区分消息类型                        │   │
│  │ 4. 错误恢复，解析失败如何处理                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓                                   │
│  输出: 结构化消息对象                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 核心挑战

| 挑战 | 描述 | 解决方案 |
|------|------|---------|
| **流式读取** | InputStream 可能读取不完整 JSON | 行缓冲 + 拼接重组 |
| **多行 JSON** | 工具调用可能跨多行 | 缓存区 + 闭合检测 |
| **标签识别** | 需要识别 chunk/thinking/tool_use 等 | 类型字段判断 |
| **错误恢复** | JSON 解析失败如何继续 | 跳过错误行 + 记录日志 |
| **性能优化** | 高频消息解析性能问题 | 零拷贝 + 对象池 |

#### 所需知识

1. **流式解析**
   - BufferedReader 逐行读取
   - 缓冲区管理
   - 字符编码处理

2. **JSON 解析**
   - Gson/Platform JSON API
   - 流式 JSON 解析
   - 错误处理

3. **协议设计**
   - NDJSON 格式规范
   - 消息类型定义
   - 版本兼容性

#### 参考实现

```kotlin
class NDJSONParser {
    private val buffer = StringBuilder()
    private val gson = Gson()

    fun parseLine(line: String): List<DaemonMessage> {
        val messages = mutableListOf<DaemonMessage>()

        // 添加到缓冲区
        buffer.append(line)

        // 尝试解析完整 JSON
        val content = buffer.toString()
        try {
            val json = JsonParser.parseString(content).asJsonObject
            messages.add(toMessage(json))
            buffer.clear()
        } catch (e: JsonSyntaxException) {
            // JSON 不完整，等待下一行
        }

        return messages
    }

    private fun toMessage(json: JsonObject): DaemonMessage {
        val type = json.get("type").asString
        return when (type) {
            "chunk" -> DaemonMessage.Chunk(json.get("content").asString)
            "thinking" -> DaemonMessage.Thinking(json.get("content").asString)
            "tool_use" -> DaemonMessage.ToolUse(
                json.get("toolName").asString,
                json.get("toolInput").asJsonObject
            )
            "complete" -> DaemonMessage.Complete(gson.fromJson(json.get("result"), QueryResult::class.java))
            "error" -> DaemonMessage.Error(json.get("error").asString)
            else -> DaemonMessage.Unknown(type)
        }
    }
}
```

#### 技术选型分析

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **纯 Swing** | • 原生 Platform API<br>• 零额外依赖<br>• 内存占用小 | • UI 开发效率低<br>• 现代化设计难 | ⭐⭐⭐ |

| **混合方案** | • 兼顾两者优势<br>• 渐进式迁移 | • 架构复杂<br>• 维护成本高 | ⭐ |

> **结论**: 建议优先使用纯 Swing，核心功能稳定后考虑 JCEF 可选集成

@Service(Service.Level.APPLICATION)
class ConfigService : PersistentStateComponent<ConfigService.State> {

    data class State(
        var cliPath: String = "",
        var apiKey: String = ""
